## Problem Statement

Рынок CS2-скинов — это рынок с информационной асимметрией. Цена предмета на маркетплейсе не отражает его реальную ценность: float value, stickers, pattern index, StatTrak — всё это создаёт premium или discount, который не виден в листинге. Покупатель должен вручную собирать данные из 3-5 источников, сравнивать, оценивать ликвидность и принимать решение за минуты (пока лот не купили). Это медленно, утомительно и error-prone.

Нужна система, которая автоматизирует сбор, нормализацию и анализ данных о конкретном предмете и выдаёт решение: **buy / no-buy / manual review** с обоснованием.

## What Makes This Cool

Это не просто price checker. Это **decision engine**, который понимает разницу между commodity-like lot (AK-47 Redline FT с float 0.25 без стикеров) и exact-premium lot (AK-47 Redline FT с float 0.150 и 4x Katowice 2014 стикерами). Для первого достаточно сравнить цену с рынком. Для второго нужно оценить premium за exact-параметры, проверить ликвидность exact-market, посчитать safe exit price.

Система должна быть как опытный трейдер, который за 30 секунд говорит "бери" или "не бери" и объясняет почему.

## Constraints

- Один пользователь, локальный запуск
- Python 3.12+ (httpx, SQLite, Pydantic, Rich)
- MVP: Steam Market + CSFloat API (Tier 1 sources)
- Rule-based logic для MVP, ML — отложено
- Бюджет на API: разумный, rate limits учтены
- Не торговый бот — не совершает сделок
- Постепенное наращивание источников
- **Credentials**: API keys хранятся в `.env` файле (python-dotenv), НЕ коммитятся в git

## Risk: Steam Market API Access

Steam не имеет официального public API для Market listings. Варианты:
- **CSFloat API** (primary) — официальный, стабильный, даёт и листинги и exact data
- **Steam Community Market** — неофициальные endpoints, rate-limited (~20 req/min), могут измениться
- **Fallback для MVP**: если Steam Market API нестабилен, использовать CSFloat как единственный source

Это **highest-risk dependency** проекта. MVP должен работать даже если Steam Market API недоступен.

## Premises

1. **Decision support > data collection.** Ценность системы — не в объёме собранных данных, а в качестве решения buy/no-buy. Если система знает мало, но принимает правильные решения — это лучше, чем система, которая знает всё, но не помогает решить.

2. **Item identity — отдельный каноничный слой.** Canonical item (weapon + skin + quality + StatTrak) — это первый уровень. Exact instance (float + stickers + pattern + kills) — второй. Pricing — третий. Смешивать их — создавать нетестируемый monolith.

3. **Два класса предметов — два пути анализа.** Commodity-like lots оцениваются по market spread и volume. Exact-premium lots оцениваются по exact-параметрам, rarity, collector demand. Один размер не подходит обоим.

4. **Rule-based MVP — правильная отправная точка.** ML требует labeled data, которых у нас нет. Rule-based система генерирует решения + accumulates labeled data для будущего ML.

5. **Liquidity = safety.** Хорошая цена на неликвидном предмете — это ловушка. Safe exit price и time-to-sell так же важны, как price advantage.

6. **Один-два источника достаточно для MVP.** Steam Market + CSFloat покрывают item identity, exact-параметры, базовые цены и историю. Добавлять Skinport/DMarket/Buff163 — v2+.

## Approaches Considered

### Approach A: Pipeline-first (RECOMMENDED)
**Summary:** Линейный pipeline: ingest → identify → enrich → price → liquidity → decide. Каждый шаг — отдельный модуль с чистым входом/выходом. CLI берёт URL или item описание, прогоняет через pipeline, выводит decision card.

**Effort:** M (Medium)
**Risk:** Low
**Completeness: 8/10**

**Pros:**
- Простая ментальная модель — данные текут в одном направлении
- Каждый модуль тестируется независимо
- Легко добавлять новые шаги без ломки существующих
- Быстрый путь до первого usable output

**Cons:**
- Расширение источников требует adapter pattern (но это управляемо)
- Линейность может ограничивать при сложных взаимозависимостях (но для MVP это не проблема)

**Reuses:** Pydantic models как контракты между шагами, httpx для всех HTTP, SQLite для кэширования и истории.

### Approach B: Plugin Architecture
**Summary:** Ядро-оркестратор + plugin system. Каждый source — plugin. Каждый analysis step — plugin. Конфигурируется через YAML.

**Effort:** L (Large)
**Risk:** Medium
**Completeness: 9/10**

**Pros:**
- Максимальная расширяемость
- Чистое разделение ответственности
- Легко включать/выключать источники

**Cons:**
- Over-engineered для одного пользователя
- Больше boilerplate до первого результата
- Plugin discovery / registration — лишняя сложность

### Approach C: Query-first (Backwards from Decision)
**Summary:** Начать с decision engine: определить buy/no-buy criteria, затем строить только те data pipelines, которые питают эти criteria. Минимальный набор данных для решения.

**Effort:** S (Small)
**Risk:** Medium
**Completeness: 6/10**

**Pros:**
- Самый быстрый путь до usable output
- Строим только нужное
- Фокус на ценности, не на инфраструктуре

**Cons:**
- Может пропустить важные данные, которые не очевидны заранее
- Сложнее расширять — нет структурного каркаса
- Тяжело тестировать — нет чистых boundaries

## Recommended Approach

**Pipeline-first (Approach A)** — лучший баланс скорости до первого результата, тестируемости и расширяемости. Для personal tool одного power user линейный pipeline с чистыми модулями — это goldilocks zone: не слишком простой (Query-first), не слишком сложный (Plugin).

### System Architecture (Sequential Pipeline)

```
CLI Interface (Rich terminal output)
       │
       ▼
Pipeline Orchestrator (runs modules in strict sequence)
       │
       ▼
┌──────────────┐
│ 1. Source     │ ← Steam Market / CSFloat API
│    Ingest     │   Failure: retry 2x, then fallback to cache or error
└──────┬───────┘
       ▼
┌──────────────┐
│ 2. Item       │ ← Resolve canonical identity
│    Identity   │   Failure: hard fail (can't proceed without identity)
└──────┬───────┘
       ▼
┌──────────────┐
│ 3. Exact      │ ← Float, stickers, pattern, StatTrak
│    Enrichment │   Failure: warn + continue as commodity (degraded mode)
└──────┬───────┘
       ▼
┌──────────────┐
│ 4. Pricing    │ ← Base price + premium calculation
│    Engine     │   Failure: return base_price only, flag as incomplete
└──────┬───────┘
       ▼
┌──────────────┐
│ 5. Liquidity  │ ← Volume, spread, safe exit
│    Analyzer   │   Failure: set liquidity=UNKNOWN, flag risk
└──────┬───────┘
       ▼
┌──────────────┐
│ 6. Decision   │ ← Buy/no-buy/review + confidence
│    Engine     │   Failure: force action=REVIEW with reason
└──────────────┘
       │
       ▼
SQLite Storage (item cache, price history, decision log)
```

**Note:** Pipeline is strictly sequential — each module depends on output of the previous. No parallel execution.

### Data Flow: Single Item Analysis

```
1. INPUT: Steam Market URL / CSFloat listing / item description
   ↓
2. SOURCE INGEST: Fetch raw data from Steam + CSFloat APIs
   → Raw listing data, inspect link, price, sticker market prices (cached in SQLite, refreshed daily)
   ↓
3. ITEM IDENTITY: Resolve canonical item
   → weapon=AK-47, skin=Redline, quality=FT, stattrak=false
   ↓
4. EXACT ENRICHMENT: Get instance-specific data
   → float=0.151, stickers=[Katowice 2014 Titan x1], pattern=661
   ↓
5. PRICING ENGINE:
   a) Base price = market median for canonical item (last 50 sales, or all if <50; minimum 5 required, else flag incomplete)
   b) Classify: commodity vs exact-premium (see Classification Rules below)
   c) If commodity: spread analysis, volume check → estimated_value ≈ base_price
   d) If exact-premium: premium calc per attribute (see Premium Heuristics below)
   → base_price=$45, estimated_value=$120, premium_reason="low float + Kato14"
   ↓
6. LIQUIDITY ANALYZER:
   → daily_volume=12, avg_spread=8%, time_to_sell="1-3 days"
   → safe_exit_price=$95 (conservative estimate)
   ↓
7. DECISION ENGINE:
   → listing_price=$80, estimated_value=$120, margin=+50%
   → safe_exit=$95, downside_risk=-15%
   → liquidity=GOOD
   → DECISION: BUY (confidence: 0.82)
   → reasons: ["underpriced by 33%", "safe exit above listing", "good liquidity"]
```

### Classification Rules: Commodity vs Exact-Premium

An item is classified as **exact-premium** if ANY of these conditions is true:
- `float_value` is in the top 5% or bottom 5% of the quality range for that wear tier
- Any sticker with market value > $5 is applied
- `paint_seed` matches a known special pattern (blue gem, fade %, etc.)
- StatTrak with float < 0.01 (factory new collector tier)

Otherwise, the item is **commodity-like**.

If pattern DB is unavailable, treat `paint_seed` as non-special and flag `pattern_premium=null` in output.

### Premium Heuristics (v1 — Rule-Based)

**Float premium:**
- Compute `float_percentile` = position within wear tier range (e.g., FT range 0.15–0.38)
- `float_factor` from tier table:
  - Top 1%: base_price × 0.50
  - Top 5%: base_price × 0.20
  - Top 10%: base_price × 0.10
  - Bottom 5% (worst): base_price × -0.10
  - Middle 80%: 0 (no premium)

**Sticker premium:**
- `sticker_premium = sum(sticker_market_price × multiplier)` per sticker
- Multipliers: Katowice 2014 Holo → 0.10–0.15, Katowice 2014 non-Holo → 0.05–0.08, other tournament Holos → 0.03–0.05, common stickers → 0 (no premium)
- Position bonus: best position (weapon-dependent) gets 1.2× multiplier
- Condition: scratched stickers get 0.5× multiplier

**StatTrak premium:**
- StatTrak items use a separate base_price bucket (median of ST sales only)
- No additional multiplicative premium — ST pricing is already reflected in base_price
- Exception: ST + low float (<0.01) gets collector premium = base_price × 0.15

**Pattern premium:**
- Lookup from hardcoded `special_patterns.json` (blue gems, fade %, marble fade)
- If pattern not in DB → pattern_premium = 0, flag for manual review

### Safe Exit Price Algorithm

```
safe_exit_price = min(
    estimated_value × 0.85,                    # 15% haircut on estimated value
    percentile_10(last_30_comparable_sales),    # 10th percentile of recent comps
)
safe_exit_price = max(safe_exit_price, base_price × 0.90)  # floor: 90% of base
```

"Comparable sales" = same canonical item within ±0.02 float range (commodity) or same canonical item + matching sticker category + float within ±0.05 (premium). If <5 comparable sales found → widen float range to ±0.05 (commodity) or ±0.10 (premium).

### Confidence Score Derivation

```
confidence = weighted_average(
    data_completeness × 0.3,   # 1.0 if all fields present, 0.5 if degraded
    price_data_quality × 0.25, # 1.0 if 50+ recent sales, 0.5 if <10
    margin_strength × 0.25,    # sigmoid(margin_pct / 20%) — stronger margin = higher
    liquidity_score × 0.2,     # 1.0 if >10/day, 0.5 if 1-10/day, 0.2 if <1/day
)
```

If confidence < 0.5 → force action=REVIEW regardless of margin.
Note: incomplete flag caps confidence at 0.49 (not 0.5) to ensure REVIEW triggers.

### Module Contracts (Pydantic Models)

```python
from enum import Enum
from pydantic import BaseModel, field_validator

class ItemClass(str, Enum):
    COMMODITY = "commodity"
    EXACT_PREMIUM = "exact_premium"

class DecisionAction(str, Enum):
    BUY = "buy"
    NO_BUY = "no_buy"
    REVIEW = "review"

class LiquidityGrade(str, Enum):
    HIGH = "high"       # >10 sales/day
    MEDIUM = "medium"   # 1-10 sales/day
    LOW = "low"         # <1 sale/day
    UNKNOWN = "unknown" # insufficient data

class Sticker(BaseModel):
    name: str
    slot: int           # 0-4
    wear: float         # 0.0 = pristine, 1.0 = scraped off

class CanonicalItem(BaseModel):
    weapon: str          # "AK-47"
    skin: str            # "Redline"
    quality: str         # "Field-Tested"
    stattrak: bool       # false
    souvenir: bool       # false

class ExactInstance(BaseModel):
    canonical: CanonicalItem
    float_value: float   # 0.151
    paint_seed: int      # 661
    stickers: list[Sticker]
    stattrak_kills: int | None = None

    @field_validator("stattrak_kills")
    def validate_stattrak_kills(cls, v, info):
        canonical = info.data.get("canonical", {})
        is_st = canonical.get("stattrak", False) if isinstance(canonical, dict) else getattr(canonical, "stattrak", False)
        if v is not None and not is_st:
            raise ValueError("stattrak_kills must be None for non-StatTrak items")
        return v

class PricingResult(BaseModel):
    canonical: CanonicalItem
    base_price: float
    item_class: ItemClass
    estimated_value: float
    premium_breakdown: dict[str, float]  # {"low_float": 30.0, "sticker": 45.0}
    incomplete: bool = False  # true if premium calc was degraded

class LiquidityResult(BaseModel):
    canonical: CanonicalItem
    avg_daily_volume: float  # avg sales/day over 30-day window (e.g., 0.1 = 3 sales/month)
    avg_spread_pct: float
    min_sell_days: int       # structured: min estimate
    max_sell_days: int       # structured: max estimate
    safe_exit_price: float
    grade: LiquidityGrade

class Decision(BaseModel):
    action: DecisionAction
    confidence: float        # 0.0 - 1.0
    listing_price: float
    estimated_value: float
    margin_pct: float
    safe_exit_price: float
    reasons: list[str]
    risk_flags: list[str]
```

## Implementation Order (MVP)

### Wave 1: Foundation (build first)
1. **Pydantic models** — все контракты между модулями
2. **SQLite storage** — schema, cache layer, CRUD
3. **Source ingest** — Steam Market API + CSFloat API clients
4. **Item identity** — canonical resolver

### Wave 2: Analysis Core
5. **Exact enrichment** — float, stickers, pattern extraction
6. **Pricing engine** — base price + commodity/premium classifier + premium calc
7. **Liquidity analyzer** — volume, spread, safe exit

### Wave 3: Decision + CLI
8. **Decision engine** — rule-based buy/no-buy logic
9. **CLI interface** — Rich-based output, argument parsing
10. **Pipeline orchestrator** — wires everything together

### Wave 4: Polish (post-MVP)
- Price history tracking
- Portfolio management
- Monitoring / alerts
- Additional sources (Skinport, DMarket, Buff163)
- ML-based pricing models

## Success Criteria

1. Ввод URL → decision card за < 10 секунд (warm cache); < 30 секунд (cold cache)
2. Ручная валидация commodity/premium классификации на 20 тестовых предметах перед релизом Wave 3
3. Estimated value в пределах ±20% от реальной цены на тех же 20 тестовых предметах
4. Safe exit price консервативен — проверяется на 10 исторических сделках (safe_exit < actual_sell_price)
5. CLI понятен без документации — smoke test: новый терминал, одна команда, результат понятен

## Failure Modes

| Module | Failure | Behavior |
|--------|---------|----------|
| Source Ingest | API timeout / rate limit | Retry 2x with backoff, then check cache. If no cache → hard fail |
| Source Ingest | API returns unexpected format | Log warning, hard fail with "source format changed" |
| Item Identity | Can't resolve canonical | Hard fail — can't proceed without identity |
| Exact Enrichment | Float/sticker data unavailable | Warn, continue as commodity (degraded mode) |
| Pricing Engine | No recent sales data | Use cached median if <7 days old, else flag incomplete |
| Pricing Engine | Pattern DB unavailable | Treat as non-pattern, flag `pattern_premium=null` |
| Liquidity | Insufficient data (<5 sales in 30d) | Set grade=UNKNOWN, force Decision to REVIEW |
| Decision Engine | Any upstream incomplete flag | Cap confidence at 0.5, prefer REVIEW |
