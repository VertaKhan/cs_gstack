## Vision

### 10x Check
10x version: система, которая не просто анализирует один лот по запросу, а непрерывно сканирует все новые листинги на всех площадках, находит underpriced лоты, ранжирует по margin × liquidity × confidence, и показывает топ-10 возможностей в real-time dashboard с push-уведомлениями. Но для MVP это ocean, не lake. MVP = один лот за раз, CLI, 2 источника. Это правильно.

Самый сильный first usable outcome: **ввёл URL лота на CSFloat → за 10 секунд получил decision card с buy/no-buy и обоснованием.** Не portfolio, не monitoring, не batch — один лот, одно решение, одна команда.

Всё остальное строится поверх этого. Если этот flow работает хорошо, остальное — incremental improvements.

## Premise Validation

### Pipeline-first: правильный ли выбор?
**Да.** Для одного пользователя с одним use case (analyze single item) линейный pipeline — оптимальный выбор. Plugin architecture добавила бы ~300 LOC boilerplate без пользы. Pipeline можно всегда рефакторить в plugin system позже, если понадобится.

### Не over-engineered ли 6-модульная система для personal tool?
**Нет, но с оговоркой.** 6 модулей — это правильная декомпозиция домена (identity ≠ enrichment ≠ pricing ≠ liquidity ≠ decision). Каждый модуль решает конкретную задачу. Over-engineering было бы, если бы модули общались через message bus или plugin registry. Здесь они просто вызываются последовательно — это функции, не сервисы.

### Что из Wave 1-3 можно убрать?
**`cs2 history` команду можно отложить.** Price history display — отдельная feature, не нужна для buy/no-buy decision на конкретный лот. Но price history *data* нужна для pricing engine (base price calculation), так что таблица в SQLite всё равно будет. Убираем только CLI команду.

## Key Architectural Decisions (locked)

1. **CSFloat as primary source, Steam as secondary.** CSFloat API is stable and documented. Steam Market API is undocumented and rate-limited. MVP должен работать даже если Steam API недоступен.

2. **SQLite for everything.** Cache, price history, decision log, config cache. Один файл, zero deployment complexity, perfect for single-user local tool.

3. **TOML for config.** Readable, writable, well-supported in Python. `.env` for API keys only (security), TOML for everything else (thresholds, preferences).

4. **Rule-based first.** Premium heuristics в DESIGN.md конкретны и тестируемы. ML — только после accumulation of labeled data from decision log.

5. **Strict sequential pipeline.** Нет параллелизма, нет async. Для single-item analysis это правильно. Async — только для future monitoring/scanning.

## Success Metrics (measurable)

1. **Time to decision**: < 10s warm cache, < 30s cold — measured in CLI output
2. **Manual validation**: 20 test items before calling MVP done
3. **Decision accuracy**: Track in decision_log, review weekly, target >70% correct calls
4. **System reliability**: No crashes on malformed input (test with edge cases)
